
# 交易复盘系统 - 数据库表结构文档

## 1. 系统概述
本系统主要围绕**加密货币交易复盘**构建，核心数据流向为：通过用户授权的 `ApiKey`，利用定时任务或主动触发获取交易所的原始 `Trade`（成交记录），再通过内部引擎将其聚合为具有完整开平仓生命周期和盈亏数据的 `Leg`（头寸/持仓），最终供用户打上 `Tag`（标签）进行高阶复盘与总结。

## 4. 关键架构设计说明

1. **不可变底稿原则**：`Trade` 表的数据一旦从交易所拉取并去重入库，即视为财务流水底稿，系统业务层不应对其进行修改。所有复杂的盈亏计算均在 `Leg` 表中体现。

2. **读写分离与冗余**：`Leg` 表中冗余了 `userId` 字段，这是为了避免在查询用户历史复盘数据时，需要执行 `User -> ApiKey -> Trade -> Leg` 这样低效的跨表多级联查。

3. **交易指纹去重**：`Trade.id` 不是交易所返回的 ID，而是系统生成的指纹。三种策略：`tid:{tradeId}` → `oid:{orderId+关键字段}` → `fp:{symbol+时间+方向+价格+数量}`。通过 `(apiKeyId, tradeId)` 唯一约束防止重复入库。

4. **净利润计算**：`Leg.netPnL = realisedPnLusd - commissionUsd + fundingFeeUsd`。每次同步重聚合时重新计算，资金费归集时也会更新。

5. **API Secret 加密**：`ApiKey.apiSecret` 使用 AES-256-GCM 加密存储，格式为 `iv:tag:encrypted`（hex）。需要 `ENCRYPTION_KEY` 环境变量（32 字节 / 64 hex 字符）。`apiKey` 本身明文存储。

6. **数据隔离现状**：用户级别数据（Leg、Tag、AiReport 等）已通过 `userId` 做了隔离。但 `Trade` 表通过 `apiKeyId` 关联，未直接关联 `userId`，API Key 下挂的 Trade 数据尚未完成用户归属隔离。
